# DUVU CALIBRATION

이 파일은 `/calibrate`의 수용 기준이다. DUVU가 단순한 토큰 묶음이나 코드 생성 유틸리티가 아니라, **AI가 세계 최고 수준의 디자인과 프론트엔드/UI/UX/비주얼을 실제 코드와 화면으로 만들어내게 하는 도구**인지 판정한다.

## 최상위 비전

DUVU의 기준은 "작동한다"가 아니다. 작동은 바닥이다.

DUVU는 사용자가 도메인, 분위기, 플랫폼, 요구사항만 말해도 AI가 다음을 수행하게 만드는 시스템이어야 한다.

- 세계 최고 레퍼런스의 표면을 베끼지 않고, 그 구조와 리듬과 미감을 분해해 재조합한다.
- 모든 색, 여백, 타이포, 반경, 모션, 생략, 강조에 방어 가능한 의도와 의미를 부여한다.
- 평균적인 AI 카드 UI, 장식 과잉, 무성격 템플릿, 예쁜 척하는 무근거 효과를 실패로 본다.
- 아름다움, 사용성, 접근성, 구현 가능성을 동시에 만족한다.
- AI가 발전할수록 더 좋은 판단을 하도록 바닥은 강제하고 천장은 열어둔다.

이 비전에 부합하지 않으면 테스트가 통과해도 `A`가 아니다.

## 판정 태도

- `PASS`: 철학, 제품 방향, 구현, 검증 증거가 함께 충족된다.
- `PARTIAL`: 방향은 맞지만 일부 증거가 약하거나 외부/수동 검증에 남아 있다.
- `FAIL`: 작동하더라도 세계 최고 수준의 디자인을 강제한다는 목적에 미달한다.
- `DEFER`: 현재 저장소만으로 검증할 수 없는 외부 레퍼런스, 권한, 벤치마크, 수동 비전 비교 영역이다.

근거 없는 낙관은 실패다. 비판을 위한 비판도 실패다. `/calibrate` 중에는 이 기준 파일을 수정하지 않는다.

## 비타협 원칙

### 1. 아름다움은 첫 관문이다

기능, 규격, 토큰 수량, 테스트 통과는 아름다움을 대체하지 못한다. DUVU 산출물은 세계 최고 레퍼런스와 나란히 놓였을 때 어색하지 않은 수준을 목표로 해야 한다.

`A` 조건:

- 화면 첫인상이 평균적인 AI 생성물처럼 보이지 않는다.
- 강한 미학 방향과 기억에 남는 구조적 장면이 있다.
- 아름다움이 장식이 아니라 정보 계층, 여백, 타이포, 리듬, 상호작용에서 나온다.

### 2. 모든 결정에는 의도와 의미가 있다

DUVU는 "대충 예쁜 값"을 허용하지 않는다. 작은 간격, 투명도, 색상, 반경, 생략에도 왜 그런지 설명 가능한 의도가 있어야 한다.

`A` 조건:

- 요소의 존재 이유, 값의 정교함, 주변 요소와의 관계가 설명된다.
- 컴포넌트와 템플릿은 사용자가 무엇을 느끼고 이해해야 하는지까지 표현한다.
- 의도 없는 추가, 삭제, 효과, 정렬, 장식은 실패로 잡힌다.

### 3. 비움과 과감함이 동시에 있다

최고 수준의 디자인은 많이 넣어서 만들어지지 않는다. DUVU는 불필요한 것을 제거하면서도, 필요한 곳에서는 과감한 배치·강조·생략을 선택해야 한다.

`A` 조건:

- 여백은 빈 공간이 아니라 설계된 리듬으로 작동한다.
- 중요한 것은 확실히 중요하게 보이고, 덜 중요한 것은 물러난다.
- 절제와 과감함이 충돌할 때도 의도 있는 선택으로 해결한다.

### 4. AI 슬롭을 구조적으로 차단한다

DUVU는 AI가 흔히 만드는 placeholder, 중첩 카드, CTA 남발, raw 값 우회, 장식용 glow/blob, 모바일 잘림, 작은 텍스트, 의미 없는 테두리를 시스템적으로 막아야 한다.

`A` 조건:

- 나쁜 산출물을 사람이 보고 나서 감상평으로 지적하는 것이 아니라, 가능한 한 생성 전/중/후 게이트에서 차단한다.
- 좋은 예시는 통과하고 나쁜 예시는 실패하는 검증이 있다.
- 실패 메시지와 문서가 AI에게 다음 수정을 유도할 만큼 구체적이다.

### 5. 디자인 철학이 시스템으로 내려온다

철학 문서와 실제 산출물이 분리되면 실패다. DUVU의 철학은 토큰, 템플릿, 컴포넌트, 데모, 스킬, 감사 규칙으로 내려와야 한다.

`A` 조건:

- 데모만 따로 꾸미는 임시방편이 없다.
- AI 스킬로 실행해도 같은 디자인 원칙이 적용된다.
- 프리셋과 템플릿은 단순 목록이 아니라 판단 재료와 제약으로 작동한다.

### 6. 접근성과 규격은 미감의 반대가 아니다

WCAG, HIG, Material Design 같은 기준은 DUVU의 미감을 제한하는 장애물이 아니라 최소 품질 바닥이다.

`A` 조건:

- 대비, 터치 타겟, motion 절제, 텍스트 가독성, 화면비 대응이 자동 또는 반복 가능한 방식으로 검증된다.
- 규격을 맞추기 위해 디자인이 평범해지는 것이 아니라, 규격 안에서 더 나은 미감을 만든다.
- 표준을 우회하거나 "예뻐 보이니 괜찮다"로 넘기지 않는다.

### 7. 세계 최고 레퍼런스를 학습하되 복사하지 않는다

DUVU는 레퍼런스를 표면 스타일로 소비하지 않는다. 좋은 사이트가 왜 좋은지 분해하고, 재사용 가능한 구조와 판단으로 보존해야 한다.

`A` 조건:

- 레퍼런스의 구조, 밀도, 리듬, 타이포, 상호작용 품질을 분석 대상으로 삼는다.
- 저작권 리스크가 있는 원본 클론과 배포물은 분리된다.
- 1:1 비전 비교가 없으면 그 한계를 정직하게 `PARTIAL` 또는 `DEFER`로 둔다.

### 8. AI 도구로서 실제 사용 가능하다

DUVU는 사람이 읽는 디자인 선언이 아니라 AI가 실제 UI/프론트엔드 코드를 만들 때 쓰는 도구다.

`A` 조건:

- 사용자는 요구사항만 말해도 AI가 적절한 미학 방향, 프리셋, 템플릿, 플랫폼 출력을 선택할 수 있다.
- 제품 정체성, CLI, 스킬, 문서, 데모가 하나의 실행 흐름으로 연결된다.
- 여러 플랫폼에서도 같은 의미 체계와 품질 기준이 유지된다.

### 9. 운영 가능한 품질 체계다

세계 최고를 목표로 하려면 감각만으로는 부족하다. DUVU는 반복 검증, 회귀 방지, 문서화, 지식 축적을 통해 품질을 운영할 수 있어야 한다.

`A` 조건:

- 회귀 테스트, 시각 검증, 보안 검증, 패키징 검증, 문서/Cortex 정합성이 유지된다.
- 중요한 설계 판단은 다음 에이전트가 이어받을 수 있게 기록된다.
- 검증 체계가 제품을 과도하게 경직시키지 않고, 더 나은 구현으로 교체될 수 있다.

## 증거 수집 방식

`/calibrate`는 위 원칙을 실제 코드와 문서와 실행 결과에 대조한다. 특정 파일명이나 명령은 기준이 아니라 현재 저장소에서 증거를 찾는 대표 경로다.

확인해야 할 증거 유형:

- 제품 비전: README, PRD, 디자인 철학, 스킬 문서가 같은 야망을 말하는가.
- 생성 계약: 토큰, 템플릿, 컴포넌트, reference가 철학을 AI가 쓸 수 있는 제약으로 바꾸는가.
- 실제 산출물: 데모와 생성 결과가 세계 최고 수준의 미감, 가독성, 반응형 품질을 보여주는가.
- 자동 검증: 회귀 테스트, 시각 감사, 토큰 lint/audit, 보안/패키징 검증이 실패를 잡는가.
- 운영 지식: doc-sys와 Cortex가 현재 상태와 결정 이유를 복원할 수 있는가.

현재 대표 명령:

```bash
npm test
npm audit --audit-level=moderate
ctx audit --fix
find bin lib -name '*.js' -print0 | xargs -0 -n1 node --check
```

동등하거나 더 강한 검증이 있으면 위 명령은 대체할 수 있다.

## A 판정 조건

프로젝트 완성도 `A`는 다음을 모두 만족할 때만 부여한다.

- 최상위 비전에 부합한다.
- 비타협 원칙 1~9가 `PASS`다.
- 자동 검증과 실제 코드 확인이 모두 통과한다.
- 발견된 `FAIL`이 없다.
- 남은 `DEFER`는 외부 레퍼런스 자산, 권한, 수동 비전 비교처럼 현재 저장소 자동 검증만으로 판정할 수 없는 항목뿐이다.

`A+`는 세계 최고 레퍼런스와의 반복 가능한 벤치마크와 실제 산출물 비교까지 통과할 때만 고려한다.
