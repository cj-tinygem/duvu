# DUVU 디자인 규칙

## 핵심 원칙

### 1. 비움의 미학
- 디자인을 잘 한다는 것은 '채우는 것'이 아니라 '비우는 것'이다
- 모든 요소에 "이게 없으면 안 되나?"를 묻는다
- 장식적 요소는 제거. 의미 있는 요소만 남긴다

### 2. 의도와 의미 시스템
- **의도(Intent)**: 설계자가 "왜 이렇게 만들었는가" — 시선 유도, 무게 배분, 여백의 이유
- **의미(Meaning)**: 사용자가 "이것이 무엇을 뜻하는가" — 행동 유도, 정보 전달, 감정
- 의도와 의미는 구분되어야 한다. 둘 다 있어야 한다
- padding 1px, 색상 1%, opacity 0.01 — 모든 값에 이유가 있어야 한다
- "이 값이 왜 이 값인가?"까지 가야 한다. 단순함은 출발점이 아니라 도착점이다
- 의도 분석 3단계: 존재(있어야 하는가?) → 정교함(왜 이 값인가?) → 관계(주변과 균형은?)

### 3. 과감한 3요소
- **과감한 배치**: 공간을 대담하게 활용. 비대칭, 크기 대비
- **과감한 강조**: 중요한 것은 확실히 강조. Primary CTA는 단 1개
- **과감한 생략**: 불필요한 것은 과감히 제거. Less is more

## 금지 목록 (NEVER)

### 시각적 금지
- ❌ 의미 없는 테두리/아웃라인/보더 남발 (ZERO borders unless absolutely necessary)
- ❌ 장식용 그라디언트, 그림자 남발
- ❌ Lorem ipsum, placeholder 이미지
- ❌ 3D 효과, 글로우 효과 남발
- ❌ 동시에 3개 이상의 폰트
- ❌ 동시에 4개 이상의 폰트 웨이트
- ❌ 16px 미만 텍스트
- ❌ 44px 미만 터치 타겟 (버튼, 링크, 입력)

### 구조적 금지
- ❌ `--duvu-` 접두사 없는 임의 CSS 변수
- ❌ 시스템 외부의 매직 넘버 (spacing, radius, color 등)
- ❌ 하드코딩된 색상값 (반드시 토큰 변수 사용)
- ❌ 의도 없는 편집
- ❌ 미리보기에서만 수정하고 시스템에 반영 안 하기

### 접근성 금지
- ❌ 본문 텍스트 대비 4.5:1 미만
- ❌ 대형 텍스트 대비 3:1 미만
- ❌ 색상만으로 정보 전달
- ❌ 포커스 표시 없는 인터랙티브 요소
- ❌ `prefers-reduced-motion` 미지원

## 레이아웃 규칙

### Spacing (시스템 토큰 기반)
- 관련 요소 그룹은 가까이 (근접성 원칙)
- 다른 그룹은 멀리 (그룹 간 spacing은 그룹 내의 2배 이상)
- 모든 spacing은 `--duvu-space-*` 토큰 또는 플랫폼별 명명 상수에서 파생 — 산출물의 임의 하드코딩 px 금지
- 히어로 주변: `--duvu-space-hero`(160px) — 타이틀에 충분한 호흡 공간
- 섹션 간: `--duvu-space-section`(96px) — 시원시원한 구분
- 통계 그룹: 숫자↔라벨은 `--duvu-space-stat-label-gap`(4px)으로 밀착, 그룹 간은 `--duvu-space-stats-gap`(64px)
- **핵심 수치는 presets.json의 layout_tokens에서 정의** — CSS/플랫폼 산출물은 토큰 또는 명명 상수를 통해 소비

### 의도 시스템 (Spacing에 적용)
- 모든 간격에 "왜 이 값인가?"를 답할 수 있어야 한다
- 숫자와 라벨이 가까운 이유: 같은 그룹이므로 (근접성)
- 그룹 간 간격이 넓은 이유: 다른 정보이므로 (분리)
- 히어로 여백이 넓은 이유: 가장 중요한 메시지이므로 (주목도)

### 카드/컨테이너
- padding: 최소 16px (compact), 기본 24px, 넉넉 32px
- 카드 간 gap: 최소 12px, 기본 16px
- 카드 내부 요소 간 gap: 최소 8px, 기본 12px
- 중첩 카드 금지 (카드 안의 카드는 surface2로 구분)
- 코드/데이터 표면은 전역 `radius-sm`을 그대로 받지 않고 전용 반경 상한 토큰을 사용한다. `pill` 스타일에서도 코드 박스가 캡슐 형태로 변형되면 실패다.

### 버튼
- min-height: 44px (HIG 필수)
- padding: 최소 10px 16px
- Primary 버튼은 화면에 1개만
- 버튼 그룹 간 gap: 8~12px
- 아이콘 버튼도 44x44 최소

### 입력 필드
- min-height: 44px
- padding: 12px 16px
- 라벨과 필드 간 gap: 6~8px
- 필드 간 gap: 12~16px

### 그리드
- 최대 4열 (데스크톱)
- 모바일 일반 콘텐츠는 1~2열, 탐색/프리셋처럼 짧은 반복 항목은 3열까지 허용한다
- singleton 마지막 줄 방지: 마지막 행에 1개만 남는 배치는 피한다. 마지막 줄이 덜 찬 것 자체가 문제가 아니라, 1개만 덩그러니 남는 경우가 문제다.
- 프리셋 칩/탭은 `flex-wrap` 방치가 아니라 균형 그리드로 행을 분산한다
- 모바일 탭은 가로 스크롤/nowrap으로 잘리게 두지 않고 3열 이하의 명시적 그리드로 접는다
- 코드 출력 플랫폼 탭은 가로 스크롤/nowrap으로 숨기지 않는다. 7개 항목은 좁은 화면에서 3/2/2처럼 균형 래핑되어 모두 보여야 한다.
- 전역 밸런스 시스템은 기존 `intent`, `meaning`, `page_patterns`, `layout.restraint`, `layout.multiplier`, `components.level`을 먼저 읽고 spacing/rhythm/alignment/wrapping을 결정해야 한다. 밸런스 규칙이 상위 철학을 덮어쓰면 실패다.
- 밸런스 시스템은 토큰을 대체하지 않는다. 토큰 값을 직접 새로 발명하는 것이 아니라, 토큰과 layout_tokens를 어떤 비율로 쓸지 결정하는 계층이어야 한다.

## 색상 규칙

### 대비 기준
| 조합 | 최소 대비 | 비고 |
|---|---|---|
| fg vs bg | 4.5:1 | WCAG AA (본문) |
| fg vs surface | 4.5:1 | WCAG AA (카드 내 본문) |
| fg2 vs bg | 2.5:1 | 보조 텍스트 (권장 3:1) |
| accent vs bg | 3:1 | WCAG AA (대형 텍스트/UI 컴포넌트) |
| btnText vs action fill | 4.5:1 | 버튼 텍스트 가독성 |
| surface vs bg | >1.05:1 | 표면 구분 가능 |

### 색상 사용 규칙
- accent 색상은 CTA/링크/활성 상태에만
- accent 과다 사용 금지 (화면 면적의 10% 이하)
- fg3는 disabled/placeholder에만
- 시맨틱 색상(success/warning/error)은 해당 상태에서만

### 다크/라이트 모드
- 모든 프리셋은 다크/라이트 동시 지원 필수
- 단순 색상 반전 금지 — 각 모드에 맞는 미세 조정
- 다크 모드 bg: 4~8% lightness
- 라이트 모드 bg: 94~98% lightness

## 타이포그래피 규칙

### 계층
- 한 화면에 최대 4단계 (제목, 소제목, 본문, 캡션)
- 제목 ↔ 본문 크기 비율: 최소 1.5배
- 줄간격: 본문 1.5, 제목 1.15~1.25
- 생성 UI의 기본 타입 토큰 하한은 `xs 16px`, `sm 16px`, `base 17px`이다.
- 섹션 라벨과 섹션 제목 사이에는 최소 20px 수준의 호흡을 둔다. 제목과 직접 붙은 작은 라벨은 실패다.
- 팔레트 카드의 이름/분류 간격은 CSS 박스 기준 6~10px, 실제 텍스트 라인 기준 2~8px 범위로 유지한다. 단, 모바일이나 compact landscape처럼 압축된 카드에서는 글리프 측정 오차를 감안해 0px에 가까운 미세 음수(`-0.5px` 이내)는 허용한다. 2줄 이름에 맞추려고 1줄 이름과 분류 사이에 가짜 공백을 넣으면 실패다.
- 팔레트 카드의 색상 swatch는 카드 높이의 30~39% 범위여야 한다. 팔레트 탐색은 색상 면보다 이름/분류를 읽고 선택하는 UI이므로 색상 면이 주역처럼 보이면 실패다.
- 팔레트 카드의 정보 영역은 실제 텍스트 라인 기준으로 이름과 분류가 2~8px 호흡을 유지해야 한다. 2줄 이름에 맞추려고 1줄 이름과 분류 사이에 가짜 공백을 넣으면 실패다.
- 같은 팔레트 그리드 안의 카드 높이는 1px 오차 안에서 같아야 한다. 짧은 이름 카드와 긴 이름 카드의 높이가 달라지면 스캔 리듬이 깨지므로 실패다.
- 팔레트 카드의 정보 영역 하단 여백은 실제 텍스트 라인 기준 17px 이하여야 한다. 통일 높이를 만든다는 이유로 텍스트 아래가 크게 비어 보이면 실패다.
- 팔레트 카드는 세로가 낮은 가로형 화면에서도 검증해야 한다. 일반 데스크톱/모바일만 통과하고 compact landscape에서 비례가 무너지면 실패다.
- 팔레트 그리드는 컬럼 수를 singleton 마지막 줄 방지만으로 정하지 않는다. 카드의 최소 읽기 폭을 먼저 보장해 이름 줄바꿈과 카드 높이 증가가 연쇄 발생하지 않게 해야 한다.
- 45개 팔레트처럼 개수가 고정된 탐색 그리드에서는 `6열의 마지막 줄 3개`와 `4열의 마지막 줄 1개`를 같은 문제로 취급하면 안 된다. 반쯤 찬 마지막 줄은 허용 가능하지만, singleton 마지막 줄은 실패다.
- 팔레트 기하 규칙은 단일 엔진에서 관리해야 한다. 화면비별 CSS 조각을 흩뿌리지 말고, 최소한 `viewport mode(short/base/tall/mobile)`와 `name line mode(single/multi)`를 입력으로 받아 카드 높이, swatch 높이, info padding, label gap을 함께 산출해야 한다.
- 좁고 낮은 화면에서는 텍스트 블록을 우선 압축하고, 높고 넓은 화면에서는 swatch 면적을 우선 확장해야 한다. 카드 전체를 기계적으로 함께 줄이거나 늘리면 실패다.
- 팔레트 엔진은 전역 밸런스 시스템의 하위 계층이어야 한다. 전역 밸런스가 정한 density/rhythm/alignment를 무시하고 별도 비례를 독자적으로 만들면 실패다.

### 폰트 선택 의도
각 폰트는 공감각적(synesthetic) 의미를 갖는다:
- **Pretendard**: 안전, 깔끔, 한국어 기본
- **Inter**: 중성적, 범용, 데이터 친화
- **DM Sans**: 기하학적, 현대적, 친근
- **Space Grotesk**: 기술적, 코드 친화
- **Satoshi**: 독특, 균형, 크리에이티브
- **General Sans**: 부드럽고 세련
- **Plus Jakarta**: 따뜻하고 현대적
- **Outfit**: 둥글고 친근, 앱 UI
- **Sora**: 기하학적 모던, 디지털 네이티브
- **Manrope**: 세미 모노, 기술적이면서 친근

## 모션 규칙

### 기본 원칙
- 모든 모션에는 목적이 있어야 한다 (피드백, 방향, 관계)
- 장식용 모션 금지
- 사용자 액션에 대한 즉각 피드백: 0.1~0.2s
- 상태 전환: 0.3~0.5s
- 페이지 전환: 0.5~0.8s

### ease 함수
- 일반 전환: `cubic-bezier(0.16, 1, 0.3, 1)` — 빠른 시작, 부드러운 감속
- 스프링: `cubic-bezier(0.34, 1.56, 0.64, 1)` — 오버슛 있는 탄성
- 표준: `cubic-bezier(0.4, 0, 0.2, 1)` — Material 표준

### 접근성
- `prefers-reduced-motion: reduce` 시 모든 animation-duration을 0.01ms로
- 필수 모션(로딩 스피너 등)만 유지
- 자동 재생 애니메이션 금지

## 품질 기준

### Aesthetic direction 게이트

DUVU가 이겨야 하는 지점은 협업/캔버스/호환 계층이 아니라 미감이다. 생성 전에 반드시 하나의 강한 미학 방향을 정하고, 색상·타이포·여백·모션·컴포넌트 선택을 그 방향에 종속시킨다.

- 모든 템플릿은 `aestheticDirection`과 `memorability` 점수를 통과해야 한다.
- 화면에는 기억에 남는 signature move가 하나 이상 있어야 한다. 예: 시네마틱 다크의 거의 비어 있는 focal plane, 동아시아 미니멀의 큰 여백과 작은 행동 하나, 오스모 툴킷의 리듬 있는 mosaic grid.
- 좋은 미감은 장식이 아니라 선택의 강도다. gradient/blob/glow를 추가하는 것은 signature move가 아니다.
- 같은 안정적 토큰을 쓰더라도 결과가 “평균적인 AI 카드 UI”로 보이면 실패다.
- 미학 방향과 토큰 정합성이 충돌하면 토큰을 우회하지 말고, 템플릿/프리셋/계약 자체를 조정한다.

### AI 슬롭 방지 게이트

DUVU는 생성형 UI가 흔히 만드는 제한된 카드 나열, 과한 그라디언트, 의미 없는 장식, 균질한 여백, 얕은 타이포 계층을 실패로 본다.

- Primary CTA는 한 viewport에 1개만 둔다.
- accent 면적은 화면의 10% 이하로 제한한다.
- 카드 안 카드, 이유 없는 보더, 장식용 글로우/그라디언트를 금지한다.
- placeholder 이미지, lorem ipsum, 무의미한 통계, 임의 아이콘 나열을 금지한다.
- 토큰 alias와 의미 토큰을 사용하고, raw hex/px 매직 넘버를 반복하지 않는다.
- 모바일에서 핵심 콘텐츠를 숨기지 않고 level이 낮은 장식 요소부터 생략한다.
- 모바일 탭/프리셋/팔레트/코드 출력 탭은 실제 렌더링에서 가로 넘침 0, 잘린 항목 0, 마지막 행 단독 항목 0을 만족해야 한다.
- 같은 입력에서 같은 품질 점수와 실패 이유가 나와야 한다.

### A+ 토큰 엔진 기준

DUVU의 A+ 기준은 “프리셋이 많다”가 아니라 토큰이 실제 생성 품질을 제한하고 증명하는가다.

- 커스텀 `--hex`도 프리셋과 같은 대비, alias, Figma parity, lint/audit 게이트를 통과해야 한다.
- 현재 프리셋은 45개 컬러, 11개 타이포그래피, 4개 레이아웃, 8개 스타일, 6개 모션, 9개 그라디언트, 17개 컴포넌트, 7개 인터랙션, 22개 템플릿 구조로 감사한다.
- Figma native DTCG import-ready 파일과 Figma REST API payload export를 분리한다.
- Figma 앱에 직접 동기화했다고 쓰지 않는다. 현재 범위는 import-ready/export-ready JSON 생성과 payload 준비다.
