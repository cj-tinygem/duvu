# DUVU 갭 분석: 현재 구현 vs 요구사항

> 현재 구현(코드)이 IDEA.md에서 도출한 요구사항과 일치하지 않는 부분을 기록한다.
> 현재 구현의 잘못이다 — DUVU는 아직 제대로 구현하지 못한 상태이다.

## 근본 진단

DUVU의 핵심 가치는 **"세계 최고의 디자인을 해체해서 재사용 가능한 조립 도구로 만드는 시스템"**이다. 현재 구현은 CLI, 프리셋 DB, 데모, 스킬 설치, 스크린샷 검증, 레퍼런스 토큰화의 기본 골격을 갖췄다. 남은 갭은 주로 레퍼런스 재현 fidelity와 실제 시각 품질을 더 강하게 증명하는 운영 루프다.

파이프라인: **Picasso(복제 수단) → DUVU(분해/저장/재조합 시스템) → AI 스킬(사용)**

---

## 심각도 범례

| 등급 | 의미 |
|---|---|
| **CRITICAL** | 핵심 가치가 누락됨. 없으면 DUVU의 존재 이유가 흔들림 |
| **MAJOR** | 세계 최고 기준에 명백히 미달. 반드시 해결 필요 |
| **MINOR** | 개선하면 좋지만, 현재 동작에 큰 문제는 없음 |

---

## CRITICAL 갭

### G-01: 템플릿이 진정한 테마가 아님 — **해결됨**
- **요구**: 템플릿을 바꾸면 페이지의 성질 자체가 바뀌어야 함
- **해결**: 17개 컴포넌트 프리셋 시스템 + 동적 프리뷰 렌더링. 22개 템플릿 모두 preview 필드(layout + cards)를 가지며, 도메인별 다른 UI 구조를 렌더링. 컴포넌트 CRUD + 초기화 지원.

### G-02: 데모 비주얼 품질이 세계 최고와 동급임을 더 강하게 증명해야 함 — **부분 해결**
- **요구**: osmo.supply, Awwwards, Dribbble과 나란히 놓았을 때 동급
- **현재**: 데모는 DUVU 토큰/프리셋을 직접 소비하고, 데스크톱/모바일 스크린샷 검증 루프가 통과한다. 다만 레퍼런스 사이트와의 1:1 비전 비교 및 템플릿별 품질 점수화는 지속 강화 대상이다.
- **다음 기준**: 모든 주요 템플릿을 스크린샷으로 축적하고, CRAP/계층/여백/색상 피로도/모션 절제 기준으로 시각 평가 기록을 남긴다.

### G-03: 시각적 검증 루프 없음 — **해결됨**
- **요구**: 코드만으로는 판단 불가. 반드시 시각적으로 보고 판단해야 함
- **해결**: Puppeteer + 시스템 Chromium으로 headless 스크린샷 캡처 → Read 도구(멀티모달)로 시각적 분석 가능. `duvu demo`를 띄우고 데스크톱/모바일 뷰를 자동 캡처하여 검증.

## MAJOR 갭

### G-04: 10개 클론 템플릿 완성도 — **부분 해결**
- **요구**: 10개 레퍼런스 사이트를 "완전히 똑같이" 재현
- **현재**: 10개 클론 계열 템플릿과 1개 Picasso 로컬 아카이브(osmo)를 토큰화했다. npm 패키지에는 저작권 리스크가 있는 `demo/clones/`를 포함하지 않고, clone 메타데이터는 로컬 전용 archive 상태를 명시한다.
- **남은 작업**: 각 클론의 실제 DOM/레이아웃/모션 fidelity를 별도 점수로 관리하고, 외부 아카이브 없이도 재사용 가능한 토큰/컴포넌트 패턴으로 더 많이 환원한다.

### G-05: CSS 외 플랫폼 코드 생성 미구현 — **해결됨**
- **요구**: 모든 플랫폼에서 코드 생성
- **해결**: CSS 직접 출력 + 7개 플랫폼 코드 템플릿 + 6개 핵심 컴포넌트의 4개 플랫폼(CSS/React Native/Flutter/SwiftUI) 프로덕션 코드. platforms.md 2455줄.

### G-06: 그리드 orphan 방지 미완성 — **해결됨**
- **요구**: 그리드 마지막 행에 빈 칸이 절대 생기면 안 됨. 시스템화 필수.
- **해결**: 22개 템플릿 전수 검사로 orphan 제거. 갤러리 count를 3의 배수로, 2col 레이아웃의 비full 카드를 짝수로 맞춤. 프로그래밍 검증으로 자동 탐지.

### G-07: 반응형 텍스트 줄바꿈 제어 — **부분 해결**
- **요구**: 의미 단위 줄바꿈, 히어로 텍스트가 모바일에서 너무 작아지면 안 됨
- **현재**: `word-break: keep-all`, `text-wrap: balance`, `layout_tokens` 기반 max-width, 모바일 스크린샷 검증을 적용한다. 문장 의미 단위별 자동 줄바꿈 알고리즘은 아직 없음.
- **IDEA.md 원문**: "'하나의 결정이 시스템을 - 만든다' 같은 식으로 되면 안되지 않나?"

### G-08: 프리셋 분류 체계의 완성도 — **해결됨**
- **요구**: 모든 플랫폼/프레임워크에 상관없이 작동하는 세계 최고의 분류
- **해결**: 6개 카테고리 + 17개 컴포넌트 + 15개 도메인 + tone/mood 메타데이터. `duvu match <domain> --tone warm|cool|neutral`로 도메인+감성 교차 매칭. 43개 컬러에 tone(warm/cool/neutral), 전 카테고리에 mood 추가.

### G-09: 차트/그래프 다양성 부족 — **부분 해결**
- **요구**: 다양한 방식의 차트/그래프
- **현재**: 스파크라인 + 바 차트 2종. class 기반 다중 차트 시스템 구축. 도넛 등 추가 가능.

### G-10: 의도 시스템의 시스템화 미완 — **부분 해결**
- **요구**: 모든 편집이 의도와 의미를 갖도록 시스템이 강제
- **현재**: SKILL.md 5단계 검증에 의도 2중 질문, 주목도/orphan/통일성 체크리스트, 컴포넌트 선택 가이드(2.5단계) 추가. 자동 강제는 아니지만 AI가 반드시 거치는 프로토콜로 구조화.

## MINOR 갭

### G-11: 메타데이터/태그 부족 — **해결됨**
- **요구**: 프리셋/템플릿에 도메인/분위기 메타데이터가 있어 AI가 자동 매칭
- **해결**: 43개 컬러, 11개 타이포, 4개 레이아웃, 8개 스타일, 6개 모션에 `domains` 배열(15개 도메인). 컬러에 `tone`(warm/cool/neutral), 전 카테고리에 `mood` 필드. `duvu match <domain> --tone` CLI 명령.

### G-12: 색상 조합 출처 투명성
- **요구**: 유명 색상 조합 사이트에서 인기 조합을 직접 가져올 것
- **현재**: 43개 컬러 프리셋은 브랜드(Toss, Stripe 등) + 자연색 기반. WCAG 대비는 통과하지만, 일부 조합의 이론적/인기 기반 출처는 더 투명하게 기록할 필요가 있음.

### G-13: durable.com 벤치마킹 — **조사 완료**
- **요구**: durable.com의 훌륭한 점을 학습하여 적용
- **조사**: AI 기반 웹빌더. 산업별 템플릿 스캐폴딩, 카드 패턴 모듈화, 즉시성 강조. DUVU의 22개 도메인별 템플릿 + 17개 컴포넌트가 이 접근법을 이미 부분 반영.

## 갭이 아닌 것 (이미 구현됨)

| 항목 | 상태 |
|---|---|
| CLI 도구화 (npm install -g) | ✅ 구현됨 |
| AI 스킬 설치 (Claude/Codex/Gemini) | ✅ 구현됨 |
| 데모 웹페이지 (duvu demo) | ✅ 구현됨 (품질은 MAJOR 갭) |
| 프리셋 추가/삭제/복원 | ✅ 구현됨 |
| WCAG/HIG/MD3 대비 검증 알고리즘 | ✅ 구현됨 (ensureContrast, readableOnAccent) |
| 43개 컬러 전수조사 (대비율+tone) | ✅ 완료. btnText 대비 3:1+ 전부 통과 |
| 다크/라이트 모드 지원 | ✅ 구현됨 |
| 6개 프리셋 카테고리 + 17 컴포넌트 | ✅ 구현됨 |
| 22개 종합 템플릿 | ✅ 구현됨 + 전수 시각 검증 완료 |
| ZERO 아웃라인 원칙 | ✅ 시스템화됨 |
| prefers-reduced-motion | ✅ 구현됨 |
