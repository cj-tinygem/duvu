## 2026-03-25: 데모 서빙 및 링크 경로 오류

### 발생한 문제들

#### 1. `file://` 프로토콜로 데모 열기
- **실수**: `file:///C:/dev/duvu/demo/index.html`로 브라우저를 열었다.
- **결과**: `fetch('/api/presets')` 및 `fetch('../data/presets.json')` 모두 CORS 정책으로 차단. 데이터 로딩 실패로 프리셋/템플릿 카드가 0개 렌더링.
- **올바른 접근**: `fetch()`를 사용하는 웹앱은 반드시 HTTP 서버로 서빙해야 한다. `python3 -m http.server` 또는 `npx http-server`로 먼저 서버를 띄우고 `http://localhost:PORT`로 접근.
- **교훈**: HTML 파일을 열기 전에 JS 코드의 데이터 로딩 방식을 먼저 확인하라. `fetch`, `XMLHttpRequest`, ES module import 등이 있으면 `file://`은 안 된다.

#### 2. WSL 서버 → Windows Chrome 접속 불가
- **실수**: WSL에서 `python3 -m http.server 8888`을 띄우고 Chrome에서 `localhost:8888`로 접속 시도.
- **결과**: `net::ERR_CONNECTION_REFUSED`. WSL2의 네트워크 격리로 Windows Chrome에서 WSL 서버에 직접 접근 불가.
- **올바른 접근**: WSL에서 서버를 띄우되 `0.0.0.0`에 바인딩하거나, `/mnt/c/` 경로에서 서버를 실행하면 Windows 쪽에서 접근 가능. 실제로 포트 9090에서 성공.
- **교훈**: WSL↔Windows 네트워크 경계를 항상 인식하라. `ss -tlnp`로 리스닝 확인 후 접근 시도.

#### 3. 데모↔클론 간 서버 루트 혼동
- **실수**: 어떤 서버가 루트인지 확인하지 않고 Clones 링크를 고쳤다.
- **현재 기준**: `duvu demo`는 `demo/` 디렉터리를 서버 루트로 삼으므로 데모 nav의 `/clones/osmo/index.html`가 맞다. 반대로 프로젝트 루트 정적 서버를 직접 띄우는 경우에는 `/demo/clones/osmo/index.html`가 맞다.
- **결과**: 서버 루트가 다른 상황을 구분하지 않으면 404 또는 프로젝트 디렉터리 목록 노출이 발생한다.
- **교훈**: 링크 경로를 설정할 때 서버 루트를 먼저 확인하라. 프로젝트 루트와 앱 루트는 다를 수 있다.

#### 4. 브라우저 캐시로 수정 미반영
- **실수**: `demo/index.html` 파일을 수정했으나, 브라우저가 캐시된 버전을 표시.
- **결과**: 수정이 반영되지 않아 "안 된다"는 사용자 피드백 발생.
- **올바른 접근**: 파일 수정 후 `eval`로 `location.reload(true)` 강제 새로고침, 또는 `navigate`로 캐시 버스팅 쿼리 추가.
- **교훈**: 파일 수정 후 브라우저 확인 시 항상 캐시를 무효화하라.

### 근본 원인 분석

- 링크를 수정하기 전에 서버 루트와 파일 구조의 관계를 파악하지 않았다
